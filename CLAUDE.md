# TripPlan — הנחיות עבודה ל-Claude

## מהות הפרויקט והסטאק

אפליקציית ווב לתכנון טיולים: משתמש יוצר פרופיל וטיולים; לכל טיול שני מצבים — **תכנון** (הצעות יעדים לפי דרישות וזמן, כולל הצעות AI, ובחירה מתוכן) ו**ביצוע** (לו"ז מסודר של היעדים שנבחרו). בפרופיל רואים את כל הטיולים.
**מטרה כפולה:** אפליקציה שימושית + פרויקט תיק עבודות ברמה גבוהה (Clean Architecture, TypeScript מלא, CI/CD).

**סטאק (החלטות סגורות):**

- Next.js (App Router) + TypeScript strict
- Supabase — PostgreSQL + Auth + RLS
- Vercel — דיפלוי אוטומטי מ-main
- GitHub Actions — type-check + lint על כל push
- TanStack Query (state מהשרת) / Zustand (state של UI בלבד)
- Zod — סכמה אחת לכל ישות, ממנה נגזר הכול
- Gemini דרך שכבת הפשטה (`src/lib/ai`), תשובות ב-Structured Output + ולידציית Zod
- shadcn/ui + Tailwind CSS לאטומים
- ריפו יחיד, ארגון קוד Feature-Based

**ישויות:** `profiles`, `trips` (עם status), `suggested_destinations`, `itinerary_items`.

## כללי הברזל של הארכיטקטורה

הפירוט המלא ב-`ARCHITECTURE.md` — הוא המחייב. תמצית:

1. **בידוד פיצ'רים.** פיצ'ר לא מייבא מהקרביים של פיצ'ר אחר — רק מ-`index.ts` שלו.
2. **אין גישת דאטה ישירה מ-UI.** קומפוננטות לא פונות ל-Supabase — הכול דרך שכבת `infrastructure/` של הפיצ'ר.
3. **סכמת Zod אחת לכל ישות.** טיפוסים, טפסים ו-API נגזרים ממנה.
4. **סודות בצד השרת בלבד.** מפתחות רק במשתני סביבה, בשימוש רק ב-`app/api` או קוד שרת. לעולם לא בקוד לקוח ולא ב-git.
5. **רוחב לפני עומק.** כל יכולת חדשה נבנית קודם כפרוסה דקה מקצה לקצה.
6. **טיול הוא מכונת מצבים.** `planning → executing → completed`. בלי דגלים בוליאניים לסטטוס.
7. **`components/ui` נשאר טיפש.** קומפוננטה שמכירה "טיול" שייכת לפיצ'ר.

מבנה התיקיות המחייב (תחת `src/`): `app/` (routes בלבד), `components/ui|layout`, `features/<feature>/{domain,application,infrastructure,components,index.ts}`, `lib/`, `db/`.

## אופן העבודה (קבוע)

- **מקורות האמת:** תמיד לעבוד לפי `PROJECT_PLAN.md` (מצב הפרויקט והשלבים) ו-`ARCHITECTURE.md` (כללי הארכיטקטורה). לפני כל משימה — לבדוק מה השלב הנוכחי ב-PROJECT_PLAN.md.
- **הסבר לפני ביצוע:** להסביר כל צעד בעברית לפני שמבצעים אותו.
- **צעד-צעד באישור:** לעבוד משימה-משימה, ולקבל אישור מהמשתמש לפני המעבר לצעד הבא.
- בסיום כל שלב/משימה — לעדכן את PROJECT_PLAN.md (סימון `[x]`, השלב הנוכחי, יומן ההחלטות).
