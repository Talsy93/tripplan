## 🎯 מהות הפרויקט

אפליקציית ווב לתכנון טיולים. המשתמש יוצר פרופיל, יוצר טיולים, ולכל טיול שני מצבים: **תכנון** (קבלת הצעות ליעדים לפי דרישות וזמן, ובחירה מתוכן — כולל הצעות מ-AI) ו**ביצוע** (לו"ז מסודר של היעדים שנבחרו לפי זמנים). בפרופיל רואים את כל הטיולים, כולל תצוגה עתידית בגוגל מפות.

**מטרה כפולה:** אפליקציה שימושית + פרויקט ברמה גבוהה לתיק עבודות (Clean Architecture, TypeScript מלא, CI/CD).

## 🏗️ החלטות ארכיטקטורה (סגורות)

| תחום            | החלטה                                   | נימוק                                              |
| --------------- | --------------------------------------- | -------------------------------------------------- |
| פריימוורק       | Next.js + TypeScript (strict)           | Frontend + Backend בפרויקט אחד, סטנדרט תעשייה      |
| Database + Auth | Supabase (PostgreSQL)                   | חינמי, Auth מובנה, RLS להרשאות, SQL אמיתי          |
| אחסון ודיפלוי   | Vercel                                  | חיבור ישיר ל-GitHub, דיפלוי אוטומטי, URL חינמי     |
| CI              | GitHub Actions                          | lint + type-check על כל push                       |
| State מהשרת     | TanStack Query                          | ניהול data fetching, cache                         |
| State מקומי     | Zustand                                 | state של UI בלבד                                   |
| ולידציה         | Zod                                     | סכמה אחת = מקור אמת לטיפוסים, טפסים ו-API          |
| AI              | Gemini (דרך שכבת הפשטה)                 | Free tier נדיב; ההפשטה מאפשרת החלפת ספק בקובץ אחד  |
| פורמט תשובות AI | Structured Output (JSON + ולידציית Zod) | לא טקסט חופשי — נכנס ישר לדאטהבייס                 |
| מבנה ריפו       | ריפו יחיד (לא Monorepo)                 | אפליקציה אחת; מונורפו = מורכבות מיותרת (YAGNI)     |
| ארגון קוד       | Feature-Based                           | קוד שמשתנה יחד גר יחד; הסטנדרט המקובל              |
| קומפוננטות UI   | שתי רמות: ui גנרי / פיצ'ר               | הרעיון מ-Atomic Design בלי הטקס; shadcn/ui לאטומים |

### מבנה תיקיות מחייב

```
src/
├── app/              # Routes ו-UI (כולל app/api — צד שרת)
├── components/
│   ├── ui/           # אטומים גנריים (לא מכירים את הדומיין)
│   └── layout/       # header, navigation
├── features/         # כל פיצ'ר = קופסה סגורה
│   └── <feature>/
│       ├── domain/         # טיפוסים, entities, סכמות Zod
│       ├── application/    # use-cases, hooks
│       ├── infrastructure/ # גישה ל-Supabase / ספקים חיצוניים
│       ├── components/     # קומפוננטות שמכירות את הדומיין
│       └── index.ts        # נקודת הכניסה היחידה של הפיצ'ר
├── lib/              # clients, utils משותפים
└── db/               # סכמת DB, migrations
```

### כללי ברזל (מונעי סלט)

הפירוט המלא בקובץ `ARCHITECTURE.md`. תמצית:

1. פיצ'ר לא מייבא מהקרביים של פיצ'ר אחר — רק מ-index.ts שלו.
2. אף קומפוננטה לא פונה ל-Supabase ישירות — הכול דרך infrastructure.
3. כל ישות = סכמת Zod אחת שממנה נגזר הכול.
4. מפתחות וסודות — רק במשתני סביבה בצד השרת. לעולם לא בקוד.
5. רוחב לפני עומק — פרוסה דקה מקצה לקצה קודם.
6. טיול הוא מכונת מצבים: `planning → executing → completed`.
7. components/ui נשאר "טיפש" — לא מכיר את הדומיין.

### מודל נתונים ראשוני (ישויות)

- **profiles** — נתוני משתמש (מקושר ל-auth של Supabase)
- **trips** — שייך למשתמש; שדות: שם, תאריכים, סטטוס
- **suggested_destinations** — יעד מוצע לטיול; שדות: שם, קואורדינטות, משך משוער, מקור (AI/ידני), נבחר/לא
- **itinerary_items** — יעד שנבחר בלו"ז; שדות: זמן התחלה, זמן סיום, סדר

## 📖 מקרא מונחים

- **Repo (Repository)** — מאגר הקוד של הפרויקט ב-Git/GitHub.
- **LTS (Long Term Support)** — גרסה יציבה עם תמיכה ארוכת טווח; תמיד מעדיפים אותה.
- **Environment Variables** — משתני סביבה; הדרך לשמור מפתחות סודיים מחוץ לקוד.
- **RLS (Row Level Security)** — הרשאות ברמת השורה בדאטהבייס; משתמש ניגש רק לנתונים שלו.
- **Middleware** — קוד שרץ לפני כל בקשה; משמש להגנה על עמודים ממשתמשים לא מחוברים.
- **Migration** — קובץ שמתעד שינוי בסכמת הדאטהבייס, כדי שהיא תהיה משוחזרת ומנוהלת בקוד.
- **CI/CD (Continuous Integration/Deployment)** — בדיקות אוטומטיות על כל שינוי + דיפלוי אוטומטי.
- **Structured Output** — הנחיית מודל AI להחזיר JSON במבנה קבוע במקום טקסט חופשי.
- **Rate Limiting** — הגבלת כמות בקשות למשתמש בפרק זמן, להגנה על המכסה החינמית.
- **State Machine** — מידול ישות כאוסף מצבים ומעברים מוגדרים (תכנון ⟵ ביצוע ⟵ הושלם).
- **Monorepo** — ריפו אחד עם כמה אפליקציות; לא רלוונטי לנו (אפליקציה אחת).
- **Feature-Based Structure** — ארגון קוד לפי תחומי פיצ'ר; ההחלטה שלנו.
- **shadcn/ui** — ספריית קומפוננטות שמעתיקה את הקוד לפרויקט; הבחירה שלנו לאטומים.
- **YAGNI (You Aren't Gonna Need It)** — לא בונים תשתית לצורך שלא קיים.
